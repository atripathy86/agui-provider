# AG-UI Provider — Session Management Design

## Problem

AG-UI protocol sends a fresh POST request for each user message. The frontend
accumulates full conversation history and sends it in `RunAgentInput.messages`
on every turn. The backend must decide how to use that history.

Agent Zero (A0) maintains its own rich internal session state: memory banks,
sub-agent context, tool output buffers, and chat log. This state cannot be
fully reconstructed from the flattened AG-UI message array.

## Chosen Design: Stateful `USER` Contexts Keyed by `threadId`

**Implementation:** [`agui_helpers/agui_server.py`](agui_helpers/agui_server.py)

A module-level dict maps each AG-UI `threadId` to a persistent `AgentContext`.
On each `POST /`, the server looks up the context by `threadId`; if not found,
a new one is created. The context is **not destroyed** after each run — it
survives until explicitly reset (via `POST /reset`) or TTL-evicted.

```
thread-abc → AgentContext(id="xK8mP2qL", type=USER, name="AG-UI · thread-ab")
thread-xyz → AgentContext(id="rT3nZ9wA", type=USER, name="AG-UI · thread-xy")
```

`AgentContextType.USER` is used (not `BACKGROUND`) so these conversations
appear in A0's native web UI chat list. A user can inspect, continue, or
interact with the same conversation from either interface.

### Properties

| Property | Value |
|---|---|
| Context lifetime | Until `POST /reset` or 2-hour idle TTL |
| Visible in A0 web UI | **Yes** (USER type) |
| History carrier | A0's internal chat log (not the AG-UI `messages` array) |
| Per-request input | Only the latest user message text is extracted |
| Concurrent contexts | One per active `threadId` |

### New Chat flow

1. UI calls `POST /api/agui-reset` with `{ threadId: "old-uuid" }`
2. Next.js route proxies to `POST /reset` on the AG-UI server
3. Server removes the context from `_thread_contexts`, calls `context.reset()`,
   `AgentContext.remove()`, and `remove_chat()` (removes persisted chat file)
4. UI generates a new UUID → passes it as `threadId` to `<CopilotKit>`
5. Next request with the new `threadId` creates a fresh `AgentContext`

## Alternative A: Stateless — Reconstruct History from AG-UI Messages

On every request, create a fresh `AgentContext` and replay the prior AG-UI
messages into A0 before passing the new user message.

**Why not chosen:**
- AG-UI `messages` is a flattened list (`user`, `assistant`, `tool` roles);
  A0's internal memory structures (memory bank, sub-agent context) are not
  representable in that format.
- Tool call results in AG-UI are strings; A0 tool results carry rich typed
  objects that inform its next decision.
- A0's reasoning about past turns is embedded in its log and memory, not just
  raw message text — replaying text loses this context.
- Performance: every turn re-initialises the full agent stack.

---

## Alternative B: `BACKGROUND` Contexts (Original Implementation)

Create a fresh `AgentContext(type=BACKGROUND)` on every request, destroy it
after the run, pass only the last user message.

**Why not chosen:**
- `BACKGROUND` contexts are explicitly filtered out of A0's web UI
  (`state_snapshot.py` line 278: `if ctx.type == BACKGROUND: continue`).
- No conversation continuity — A0 has no memory of prior turns.
- Each request is completely stateless on the agent side even though the
  frontend sends full history.

This was the original implementation and was effectively a single-turn
question-answer system despite being connected over a conversational protocol.

---

## Alternative C: Persist `threadId → context_id` to Disk

Store the mapping in a JSON file so it survives A0 process restarts.

**Trade-offs:**
- Pro: conversations survive container restarts.
- Con: `AgentContext` objects are in-memory only; after restart the mapping
  points to non-existent contexts, requiring re-creation anyway.
- The A0 `persist_chat` mechanism already saves context logs to disk and
  `chat_load.py` can reload them, but that path requires explicit load API calls.
- Not implemented now; worth revisiting if long-lived session persistence
  across restarts becomes a requirement.

---

## TTL Eviction

Idle contexts (no activity for `CONTEXT_IDLE_TTL_SECONDS = 7200`, 2 hours)
are evicted by a background coroutine (`_ttl_eviction_loop`) that wakes every
10 minutes. On eviction, the same teardown sequence runs as for explicit reset:
`context.reset()` → `AgentContext.remove()` → `remove_chat()`.

The TTL means the server is self-cleaning without requiring clients to always
call `/reset` on tab close (which browsers may not do reliably).

---

## Key Identifiers

| ID | Scope | Generated by | Lifetime |
|---|---|---|---|
| `threadId` | AG-UI conversation | CopilotKit frontend | Until page refresh or New Chat |
| `runId` | Single request/turn | CopilotKit frontend | One SSE stream |
| `context.id` | A0 internal context | A0 `AgentContext` | Until reset or TTL |
| `run_id` (event bus) | Extension callbacks | AG-UI server | Duration of one run |

The `context_id → run_id` mapping in `_context_run_map` is needed because A0
extensions receive `self.agent.context.id` but need the AG-UI `run_id` to emit
events to the correct SSE subscriber queue.
